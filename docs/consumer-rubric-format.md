# Consumer Rubric Format Specification

This document defines the format for consumer rubrics generated by SUDD's code-analyzer pipeline. persona-browser-agent's scorers parse this format to evaluate per-page, per-criterion results.

## Overview

A consumer rubric is a Markdown file with per-page sections. Each page section has three priority levels: Must Pass, Should Pass, and Deal-Breakers. Criteria are specific, testable statements derived from the project's specs, design, and code.

## Format

The rubric follows this structure:

    # Consumer Rubric: {change_id}

    Generated from: specs.md, design.md, codebase analysis
    Change scope: {description of what the change covers}

    ## {Page Name}
    Identified by: {how to identify this page — URL pattern, content, or feature}

    ### Must Pass
    - {Specific, testable criterion}
    - {Another criterion with codeintel reference}
      (codeintel: path.to.relevant.data)

    ### Should Pass
    - {Criterion that's important but not blocking}

    ### Deal-Breakers
    - {Criterion that, if failed, means instant FAIL for this page}

## Rules

### Page Identification

Use content/feature description, not URL. SPAs may not change URL between views.

**Good:** "Page with signup/registration form"
**Bad:** "/register"

### Priority Levels

**Must Pass** — Core functionality that defines the feature. If any fails, the feature is not working as specified. Scored at 100% weight in SUDD's formula.

**Should Pass** — Important quality attributes. Failure is a real issue but doesn't block the feature. Scored at 50% weight in SUDD's formula.

**Deal-Breakers** — Fundamental failures that override all other scoring:
- At HIGH confidence evidence → instant FAIL (score = 0)
- At MEDIUM confidence evidence → -20 penalty (not instant fail)
- Use sparingly — only for things that make the feature unusable

### Specificity

Every criterion must be verifiable from browser observation (text descriptions + screenshots + network log). Never reference:
- Database state
- Server logs
- Internal variables
- Other non-observable data

### Codeintel References

Criteria derived from code analysis should note the source for traceability:

    - Email format validation happens inline
      (codeintel: registration.elements.forms[0].fields[1].validation)

This enables the post-failure codeintel audit (Risk 3 mitigation) — if a criterion fails due to wrong codeintel, the reference tells the developer exactly what to check.

### Adversarial Review

Before use, consumer rubrics are adversarially reviewed by the rubric-adversary agent (2 iterations). The adversary checks:
1. **Specificity** — are criteria specific enough to catch real bugs?
2. **Completeness** — do criteria cover all features in specs/objectives?
3. **Testability** — can every criterion be verified from browser observation?
4. **Priority correctness** — are Must Pass/Should Pass/Deal-Breaker classifications correct?
5. **Contradiction check** — do any criteria contradict each other or codeintel?
6. **Over-specification** — would any criteria false-fail on valid implementations?
